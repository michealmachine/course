## 一、项目一句话介绍

这是一个面向在线教育场景的前后端分离在线课程平台，支持用户认证、机构入驻、课程管理、媒体资源管理、题库与题组、课程审核、学习记录、订单支付和统计分析等核心功能，目标是模拟一个较完整的在线课程业务系统。

---

## 二、项目背景与选题原因

### 1. 为什么做这个项目

我选择做在线课程平台，主要有三个原因：

1. **业务场景比较完整**：
   它不是单纯的增删改查，而是同时包含用户、机构、课程、媒体、审核、学习、支付等多个业务域，能够体现系统设计能力。

2. **能覆盖较多后端核心知识**：
   这个项目会自然涉及认证授权、数据库设计、缓存、文件存储、接口设计、异常处理、权限控制、异步任务、对象存储、分片上传等问题。

3. **前后端协作空间大**：
   在线课程平台既适合做后端业务系统，也适合补充前端页面、状态管理、接口联调、用户交互流程，因此更能体现完整项目经验。

### 2. 项目希望解决什么问题

从业务角度看，这个平台主要服务三类角色：

- **普通学习用户**：注册、登录、浏览课程、购买课程、学习课程、做题、查看学习记录。
- **教育机构用户**：上传课程内容、管理媒体资源、组织章节小节、管理题目和题组。
- **平台审核/管理员**：审核机构入驻、审核课程内容、管理用户角色、维护平台基础数据。

因此，这个项目的目标不是做一个简单的网站，而是尝试把在线教育平台中常见的关键链路都串起来。

## 三、项目技术栈

### 1. 后端技术栈

- **Java 17**：作为主要开发语言。
- **Spring Boot 3**：作为后端应用框架。
- **Spring MVC**：实现 RESTful API。
- **Spring Security**：实现认证授权。
- **JWT**：实现无状态登录。
- **Spring Data JPA**：处理数据库访问。
- **MySQL / H2**：开发与测试环境数据库。
- **Redis**：缓存验证码、Token 黑名单、上传状态、部分统计数据。
- **MinIO / S3 兼容接口**：存储媒体文件。
- **Thymeleaf + Mail**：生成并发送邮箱验证码、审核通知等邮件。
- **EasyExcel**：支持题目导入。
- **OpenAPI / Swagger**：生成接口文档。
- **支付宝 SDK**：支持订单支付流程。

### 2. 前端技术栈

- **Next.js 15 + React 19 + TypeScript**：构建前端应用。
- **Tailwind CSS + Radix UI / shadcn/ui**：实现界面组件。
- **Zustand**：管理客户端状态。
- **Axios**：封装 HTTP 请求。
- **React Hook Form + Zod**：实现表单验证。
- **Recharts / Nivo**：做统计图表展示。

### 3. 为什么这样选型

如果老师问“为什么用这些技术”，可以这样回答：

- 我希望项目尽量贴近主流企业开发方式，所以后端使用 Spring Boot 技术栈，前端使用 React/Next.js 技术栈。
- Spring Boot 能快速搭建分层清晰的业务系统，适合做认证、权限、事务、JPA 等完整后端能力。
- Next.js + TypeScript 则比较适合做组件化和路由组织，适合中后台和平台类项目。
- Redis、对象存储、JWT、分片上传这些能力，可以让项目从“普通课程管理系统”升级为“更接近真实业务”的系统。

## 四、项目整体架构设计

### 1. 架构风格

整体采用的是**前后端分离架构**。

- 前端负责页面展示、状态管理、表单交互、调用后端接口。
- 后端负责业务逻辑、权限控制、数据持久化、文件管理、审核流程、订单流程等。

后端内部采用的是典型的**分层架构**：

- **Controller 层**：接收请求、校验参数、返回统一响应。
- **Service 层**：承载核心业务逻辑。
- **Repository 层**：负责数据库访问。
- **Model 层**：包括 Entity、DTO、VO、枚举等。
- **Security 层**：负责 JWT 和认证授权。
- **Config 层**：负责数据库、缓存、对象存储、OpenAPI 等配置。

### 2. 为什么采用分层架构

采用分层架构有三个好处：

1. **职责清晰**：接口处理、业务逻辑、数据访问分开，代码更容易维护。
2. **便于扩展**：以后增加审核、支付、统计等模块时，可以在现有结构上继续演进。
3. **便于面试表达**：分层架构是后端项目中很典型的设计方式，容易向老师解释自己的设计思路。

### 3. 统一响应与异常处理

为了让前后端联调更稳定，我在接口层设计了统一返回格式：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

这样做的好处是：

- 前端判断成功失败更统一。
- 错误信息更容易规范化。
- 面试时可以体现我对接口可维护性的考虑。

## 五、核心业务模块回顾

## 1. 用户认证与权限管理模块

这是整个系统的基础模块，主要包括：

- 用户注册
- 登录
- 图形验证码
- 邮箱验证码
- 刷新令牌
- 注销
- 当前用户信息获取
- 角色与权限管理

### 设计思路

- 登录认证采用 **JWT**，避免服务端 Session 状态过重。
- 注册和登录前加入图形验证码，降低恶意请求风险。
- 注册和邮箱更新加入邮箱验证码，增强身份真实性。
- 接口权限通过 Spring Security 和 `@PreAuthorize` 控制。

### 这个模块体现的能力

- 我理解了认证流程和权限控制的基本原理。
- 我不是只会写登录接口，而是把验证码、刷新令牌、注销、用户信息获取等完整链路串起来了。

---

## 2. 机构管理模块

这个模块模拟的是“教育机构入驻平台”的业务流程，主要包括：

- 机构提交入驻申请
- 查询申请状态
- 审核员审核机构申请
- 审核通过后生成机构注册码
- 机构成员通过注册码注册
- 机构成员管理

### 为什么要做这一层

很多课程平台不是个人单独上传课程，而是由机构统一管理内容。加入这一层之后，系统就从“单用户课程系统”变成了“多角色、多组织协作平台”，业务复杂度会上一个台阶。

### 面试表达重点

如果老师问“这个模块有什么价值”，可以回答：

- 它引入了多角色和组织关系。
- 不同用户不再只是普通用户和管理员，而是扩展到机构、审核员等角色。
- 系统需要处理申请、审核、注册码、成员归属、机构资源管理等问题，因此更贴近真实平台业务。

---

## 3. 课程管理模块

这是平台的核心业务模块，主要包括：

- 课程创建与编辑
- 分类和标签管理
- 章节与小节管理
- 课程内容组织
- 课程审核流程
- 课程发布与下线
- 课程预览

### 我的设计理解

课程不是一张表就能完成，它是有层次结构的：

- 课程（Course）
- 章节（Chapter）
- 小节（Section）
- 小节资源（SectionResource）
- 小节题组（SectionQuestionGroup）

这样设计有两个好处：

1. **结构更清晰**：课程内容可以层层组织。
2. **扩展性更好**：一个小节既可以挂媒体资源，也可以挂题组，后续还能扩展试学、作业、学习进度等能力。

### 面试表达重点

这里可以体现我对“业务抽象”的理解：

- 我没有把课程内容简单堆在一张表里，而是拆成课程、章节、小节、资源、题组等多个对象。
- 这种设计虽然开发量更大，但更符合真实教育平台的内容组织方式。

## 4. 媒体资源管理模块

这个模块是我认为项目里比较有亮点的一部分，主要包括：

- 媒体资源元数据管理
- 文件上传
- 大文件分片上传
- 上传状态追踪
- 断点续传
- 访问 URL 获取
- 配额检查与使用统计

### 为什么这个模块是亮点

很多学生项目只做到“上传一个文件到服务器”。
但在在线课程场景里，视频往往比较大，如果只做普通上传会遇到：

- 上传时间长
- 大文件失败率高
- 中断后要重传
- 无法管理配额

### 我做了什么设计

- 使用对象存储而不是直接把文件放在本地服务器目录。
- 引入 S3 兼容分片上传机制。
- 上传过程用 Redis 记录状态，支持续传。
- 上传前检查机构是否还有剩余配额。
- 上传完成后更新配额使用量。

### 面试回答模板

如果老师问“你项目里最有技术含量的模块是什么”，我建议优先讲这个模块。
因为它比普通 CRUD 更能体现：

- 对真实业务问题的理解
- 对对象存储和分片上传的理解
- 对系统可靠性的考虑

## 七、我认为项目中的技术难点

下面这些问题很适合在复试里讲，因为老师会觉得你是真的做过，而不是只会背八股。

## 1. JWT 认证链路设计

### 难点

JWT 看起来只是“登录后发个 token”，但真实实现时要考虑：

- 登录后的 token 里放什么信息
- 如何刷新 token
- 注销后如何失效
- 如何和 Spring Security 集成
- 如何把用户角色和认证信息传递给后续请求

### 我做的解决思路

- 使用访问令牌 + 刷新令牌模式
- 使用过滤器在每次请求时解析 token
- 使用黑名单机制处理已注销 token
- 基于角色做接口权限控制

### 复试回答关键词

- 无状态认证
- 减轻服务端 Session 压力
- 与 Spring Security 集成
- 刷新令牌延长登录体验

---

## 2. 验证码与邮箱验证流程设计

### 难点

如果只有用户名密码，系统很容易被恶意注册或恶意尝试登录。
而邮箱验证和图形验证码会带来新的流程复杂度。

### 我的处理方式

- 图形验证码用于登录、注册、找回密码前的基础防刷。
- 邮箱验证码用于注册和邮箱更新场景。
- 使用 Redis 保存验证码并设置过期时间。
- 邮件发送使用模板，增强可维护性。

### 体现出的能力

这个模块体现的是：

- 我考虑了系统安全性和流程完整性。
- 我理解了缓存、验证码、邮件服务在业务系统中的协同方式。

---

## 3. 大文件分片上传与断点续传

### 难点

视频等资源较大，普通上传方式容易失败，而且用户体验很差。

### 解决思路

- 使用对象存储的分片上传能力
- 使用 Redis 保存上传状态
- 支持中断后继续上传
- 上传前检查配额，上传后更新配额

### 为什么这是亮点

这个问题已经超出了普通表单提交的层面，属于更贴近真实生产场景的问题。
如果老师追问技术点，这一块是很好的展开点。

## 十三、知识点速记卡片（帮助快速复习）

这一部分适合在复试前快速翻看，用来把项目中的核心技术点和业务点串起来。

### 1. JWT 是什么，为什么在项目里使用

- JWT 是一种无状态认证方式。
- 用户登录成功后，后端生成 token，前端后续请求携带 token。
- 后端不需要像传统 Session 那样保存大量会话状态，更适合前后端分离项目。
- 在我的项目中，JWT 主要用于登录认证、接口访问控制和用户身份传递。

### 2. Redis 在项目里的作用

可以直接概括为四类：

1. 存验证码
2. 存 token 黑名单
3. 存上传状态
4. 支撑部分统计与缓存场景

如果老师继续追问，就补一句：

> Redis 的价值不只是“快”，更重要的是它适合存放带过期时间、读写频繁、结构相对简单的数据。

### 3. 为什么课程要拆成课程、章节、小节

- 因为课程内容本身就是分层组织的。
- 如果不拆分，后面做资源挂载、题组挂载、试学、学习进度、审核流都会很别扭。
- 拆分之后，课程结构更清楚，也更容易扩展。

### 4. 为什么要用对象存储，而不是本地文件系统

- 媒体资源尤其是视频文件通常比较大。
- 本地文件系统在扩展性、可靠性、访问控制和后续扩容方面都不如对象存储合适。
- 对象存储更适合和分片上传、预签名 URL、权限控制等能力结合。

### 5. 为什么要做分片上传

- 视频文件大，普通上传失败率高。
- 一旦中断，整文件重传成本太大。
- 分片上传可以断点续传，也更接近真实生产场景。

### 6. RBAC 在项目中的作用

- RBAC 就是基于角色的访问控制。
- 不同角色访问不同接口。
- 在我的项目里，普通用户、机构用户、审核员、管理员的权限边界不同，所以需要 RBAC。

### 7. 什么是“业务闭环”

如果老师问你“项目是不是只是管理后台”，你可以说：

> 不是，这个项目不仅有基础信息管理，还有从机构入驻、课程创建、资源上传、审核发布、用户学习、订单支付到学习统计的完整链路，所以它形成了业务闭环。

## 十五、回答时容易犯的错误与规避方式

### 1. 错误：上来就背技术定义

比如老师问 JWT，你直接开始背：

- header
- payload
- signature

这样不一定错，但容易显得“像背书”。

**更好的方式：**
先说你项目里为什么用 JWT，再补原理。

### 2. 错误：把所有模块都讲得一样重

如果你什么都讲，反而会显得重点不突出。

**更好的方式：**
主讲 2~3 个你最熟的核心模块，例如：

- 认证授权
- 媒体资源与分片上传
- 课程内容结构设计

其他模块知道作用即可。

### 3. 错误：把自己说成“全都负责，什么都精通”

复试里最怕老师继续深挖，你一旦说得太满，就容易被问住。

**更好的方式：**
如实表达：

- 哪些是你重点负责的
- 哪些是你参与联调或辅助完成的
- 哪些你理解整体流程，但没有做到特别深

这样反而更真实。

### 4. 错误：只讲“做了什么”，不讲“为什么这样做”

如果你只说：

- 我写了登录
- 我写了上传
- 我写了支付

老师很难判断你的思考深度。

**更好的方式：**
每讲一个模块，都补一句：

- 为什么要这样设计
- 解决了什么问题
- 替代方案是什么
- 如果继续优化会怎么做

### 5. 错误：把项目说成“已经完全生产可用”

老师通常一听就知道学生项目和生产系统之间还有差距。

**更好的方式：**
承认不足，但体现思路：

> 这个项目已经具备了完整的业务原型和较清晰的架构，但如果继续走向生产环境，我还会进一步完善权限、安全、测试、部署和监控体系。

### 6. 错误：答题没有层次，想到哪说到哪

**更好的方式：**
每次回答尽量按这个顺序：

1. 场景
2. 问题
3. 方案
4. 效果
5. 后续优化

这样会显得逻辑非常清楚。
